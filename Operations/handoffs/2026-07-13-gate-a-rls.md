This is the last pre-client CODE gate: it role-gates the RLS cluster so that when the first client-role user is invited, that user cannot read/write any sibling client's secrets/cost/config/leads. **0 client-role users exist today, so nothing was live-exposed; this is a correctness gate for the apply, verified with a throwaway client-role probe.**

## What shipped (all applied live + verified read-only)

**3 migrations (Mgmt API, each dry-run in a rolled-back txn first, then committed):**
- `20260713120000_gate_a_rls_role_gate.sql` — the core. `clients` command-split (SELECT/INSERT/DELETE agency-role-gated; UPDATE = agency OR `client_own_clients_update` own-row) + a `BEFORE UPDATE` guard trigger `guard_client_clients_update` that freezes `subscription_status` + the 4 bundled infra keys (retell/openrouter/openrouter_management/ghl) for client-role writers. Role-gated `credentials` / `openrouter_usage_cache` / `unipile_accounts` (agency-only) + `agencies` UPDATE. Two-policy split (agency role-gated FOR ALL + client-own FOR ALL) on the 7 tenant parents (`client_custom_fields`, `lead_ai_columns`, `lead_tags`, `prompt_chat_threads`, `prompt_docs`, `prompt_versions`, `setter_ai_reports`) **and `leads`** (LEADS-ROLE-SPLIT-1, added to scope). Children (`prompt_chat_messages`, `lead_ai_values`, `lead_tag_assignments`) inherit the tightened parent scope, left as-is.
- `20260713130000_gate_a_clients_public_definer.sql` — **CATCH #1.** `clients_public` was `security_invoker=on`, so with base `clients` SELECT now agency-only a client-role user would read **nothing** through it (every client read breaks). Recreated it `security_definer` (runs as owner `postgres`, so the `has_*` expressions still resolve and no raw secret is projected) with a tenant-scoping `WHERE` (service_role → all; agency → its agency's clients; client → its own row; anon → none). `CREATE OR REPLACE` preserved the grants. Generated from the live viewdef for exact column fidelity.
- `20260713140000_gate_a_clients_client_select_revoke.sql` — **CATCH #2.** A client-own UPDATE needs a SELECT policy to LOCATE the row (`UPDATE ... WHERE id=<own>`); with no client SELECT policy the client's own-row UI-state writes silently matched 0 rows. Added `client_own_clients_select` (own row) + REVOKEd the table-level SELECT from `authenticated` and re-GRANTed column-level SELECT on the **111 non-secret columns** only (so a client can read its own non-secret row but never a secret value — its own or a sibling's). Safe because `clients_public` is now definer (unaffected by the REVOKE) and no browser path SELECTs base-clients secret columns (all secret reads are service-role edge fns).

**4 edge fns (deploy_single_fn.mjs, all ACTIVE):**
- `fetch-thread-previews` v10→**v11**, `twilio-list-numbers` v10→**v11**, `supabase-project-usage` v10→**v11** — RLS-GATE-SIBLING-1: swapped the user-JWT `clients` RLS-gate for `resolveClientAccess` (pins a client-role caller to its own client). `supabase-project-usage` also moved its secret read to a service-role client (the user client would now be denied).
- `get-openrouter-usage` v1→**v2** — **CATCH #3.** RLS-ORUSAGE-1 had a second vector: the fn used `authorizeClientRequest` (allows client-role), so a client could read BFD's bundled OpenRouter margin for its own client_id. Now `resolveClientAccess` + assert `role==='agency'` (browser-only caller, so no service-role path lost).

**Frontend (tsc clean; deploys on push via Railway):** `useTickerStats` + `SystemTicker` + `ClientLayout` — role-branch the `openrouter_usage_cache` read off for client-role users (SystemTicker renders in the shared ClientLayout, so a client would otherwise fire a now-denied read every 60s; the OPENROUTER_BALANCE stat is hidden for clients).
