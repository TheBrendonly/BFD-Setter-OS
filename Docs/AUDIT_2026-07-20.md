# AUDIT 2026-07-20 — bfd-setter

Audit of **`/srv/bfd/Projects/bfd-setter`**, the original Supabase + Trigger.dev implementation
(GitHub `TheBrendonly/BFD-Setter-OS`, Forgejo `origin`). Not the `bfd-setter-v2` rebuild.

Method: code, config, migrations, CI and git history treated as ground truth; every documentation claim
verified against them. Branch `chore/docs-audit-2026-07-20`, three commits, nothing pushed, no
application code changed.

---

## 1. TLDR

1. This is `bfd-setter`, the original. The orienting brief was accurate: Supabase (Postgres + 97 Deno
   edge functions) with the cadence engine on Trigger.dev, and it is genuinely live and working.
2. Documentation health was **better than typical and worse than it looked**: the tracking system is
   excellent and current, but the descriptive docs had quietly drifted by months.
3. **The single most important finding is a supply-chain-of-truth problem, not a content problem:**
   seven deliberately-deleted docs were resurrected by Syncthing on 2026-07-17 and were sitting at the
   more obvious paths, so `Docs/RUNBOOK.md` shadowed the canonical `SOP/RUNBOOK.md` with a copy six
   weeks older. Any grep, any fresh AI session, would have read the stale one.
4. The most dangerous single doc: `GATE_A_RLS_DRAFT_2026-07-08.md` still said "DRAFT, NOT APPLIED, do
   NOT apply unattended" for a migration applied and 24/24-verified on 2026-07-13.
5. Verification guidance was itself wrong: the repo has **no root `tsconfig.json`**, so the documented
   `npx tsc --noEmit` does not no-op, it errors and prints help. Three of four code surfaces are
   typechecked by nothing at all.
6. Tests are real and better than the repo advertises: **453 unit tests, all passing, ~10s.** Typecheck
   fails with 21 pre-existing errors under the only config that checks anything.
7. Docs classified: **6 CURRENT, 14 STALE-FIXABLE (all fixed), 8 SUPERSEDED/DUPLICATE (archived), 68
   HISTORICAL (bannered), 2 ASPIRATIONAL (relabelled), 6 MISSING.**
8. Nothing in the code blocks the first paying client. What blocks it is manual provisioning and
   Brendan's own live-test backlog.
9. Biggest structural risk to the business, not the code: **the bus factor is one, and the system's
   real state lives in Brendan's head plus 150 handoff files.**
10. **Honest one-line state:** a working, live, single-tenant-in-practice product that is
    code-complete for its first paying client and gated almost entirely on manual onboarding steps,
    live behavioural testing, and one person's availability.

## 3. Ground truth reference

Everything in this section was verified this session.

**Stack.** React 18.3 + Vite 8.1 + Tailwind + shadcn/ui frontend; 97 Deno edge functions plus a
`_shared/` module directory on Supabase (`bjgrgbgykvjrsuwwruoh`); 15 Trigger.dev tasks
(`proj_fdozaybvhgxnzopabtse`); Postgres on Supabase. Node engine pinned `>=20 <23`.

**Repo layout trap.** Edge functions and migrations live under **`frontend/supabase/`**. The root
`supabase/` holds only three loose `.sql` files and no functions and no migrations.

**Entry points.** Frontend `frontend/index.html` → `src/main.tsx` → `App.tsx`, which declares **88
`<Route>` elements** over 72 page components, guarded by `ProtectedRoute` / `AgencyRoute` /
`ClientRouteGuard` / `CreatorRouteGuard`. Every edge function is its own endpoint at
`{SUPABASE_URL}/functions/v1/<name>`; there is no router.

**Background jobs.** 7 scheduled Trigger.dev tasks, all cron-registered in source:
`analyze-sms-conversations`, `refresh-cadence-funnel`, `nudge-cold-reply`, `poll-retell-drift`,
`synthetic-probe` (all `0 * * * *`), `error-digest` (`0 22 * * *`), `weekly-client-report`
(`0 23 * * 0`). Plus 8 event-triggered tasks: `process-messages`, `process-setter-reply`,
`run-engagement`, `send-followup`, `run-ai-job`, `execute-workflow`, `place-outbound-call`,
`schedule-callback`. Caveat in source (`trigger/errorDigest.ts:11-13`): declarative crons may not
auto-register on deploy. `SESSION_PLAN` records this resolving on the 2026-07-12 deploy with all 7
registered, but it is worth re-checking after every Trigger deploy.

**Webhook receivers and auth.** Verified per endpoint:
- **Fail closed:** `stripe-webhook` (`index.ts:83-84`), `receive-twilio-sms` (`:487-501`),
  `twilio-status-webhook` (`:147`), `twilio-inbound-sms`, `receive-dm-webhook` (`auth.ts:9-14`),
  `intake-lead` (`:313`), `voice-booking-tools` (`:143`).
- **Verify-if-present, therefore forgeable while the per-client secret is NULL:** `retell-call-webhook`
  (`:141`), `retell-call-analysis-webhook` (`:209`), `retell-inbound-webhook` (`:107`),
  `bookings-webhook` (`:193`), `sync-ghl-contact` (`:352-354`), `ghl-tag-webhook` (`:403-405`),
  `workflow-inbound-webhook` (`:98-107`).
- `retell_webhook_secret` is deliberately not auto-generated (`webhook-manifest/index.ts:13`); arming
  it is a first-client gate. The Unipile scheme is a static-token compare whose header is
  **unconfirmed against Unipile's live config** (`_shared/verify-webhook.ts:76-86`).
- Authorization is centralised: `_shared/authorize-client-request.ts` (48 functions) and
  `_shared/assert-client-access.ts` (18). `verify_jwt = false` is declared for 32 functions in
  `frontend/supabase/config.toml` — whose `project_id` is still the placeholder
  `"YOUR_SUPABASE_PROJECT_ID"` (line 1).

**Schema.** 387 migrations, earliest `20250816202339_*`, latest
`20260713140000_gate_a_clients_client_select_revoke.sql`. `types.ts` declares **89 tables** and 1 view
(`clients_public`). **`types.ts` is behind the migrations**: `twilio-status-webhook/index.ts:159`
writes to `sms_delivery_events`, which `types.ts` does not declare. Migration timestamps are nominal,
not a reliable audit trail.

**Deploy reality.** No Dockerfile, no `railway.json`, no `nixpacks.toml` anywhere. Railway auto-detects
the Node build, runs `vite build`, and serves `dist` via `npx serve dist -s` using
`frontend/public/serve.json`. Frontend auto-deploys on push to `main`. Edge functions deploy via the
bundle scripts in `scripts/`, **not** the plain Supabase CLI, which silently drops `_shared/` imports.
Trigger.dev deploys via CLI. No deploy is automated by CI.

## 4. Rethink

### 4.1 Does the structure still make sense?

Mostly yes, with two real problems and one cosmetic one.

**The nesting is wrong and it costs you every session.** Edge functions and migrations live under
`frontend/supabase/` while the root `supabase/` holds three loose SQL files and nothing else. The
backend is nested inside the frontend for no reason other than Lovable scaffolding history. Both
ground-truth agents independently flagged this as their first finding. It is the single most reliable
source of wrong paths in this repo. **Recommendation: leave it.** The rename would touch every deploy
script, every doc, and every operator habit, to fix a naming aesthetic. Document it loudly instead,
which is now done in three places.

**Accidental complexity that is worth naming:**
- **Twilio signature verification is triplicated**, not shared: `receive-twilio-sms/index.ts:259-274`,
  `twilio-status-webhook/index.ts:140-153`, `twilio-inbound-sms/index.ts:11-30`. Retell and Unipile
  correctly use `_shared/verify-webhook.ts`. Three copies of a security primitive is three places to
  fix a flaw. **Highest-value small cleanup in the repo.**
- **`twilio-inbound-sms` is dead** and documented as legacy in `WEBHOOKS.md`, but still deployed and
  still reachable. Dead code with a signature-verification surface.
- **`frontend/src/pages/_archived/`** holds live-reachable-by-import webinar components
  (`WebinarSetupEnhanced`, `WebinarAnalyticsEnhanced`, `WebinarSetupLinks`) that the roadmap already
  called orphans in May.
- **`unipile-webhook` vs `receive-dm-webhook`** are two paths for the same job, and `WEBHOOKS.md`
  literally tells the operator to "check before pointing new accounts here vs" the other. That is an
  unresolved decision wearing a documentation costume.

**Modules doing too much:** `executeWorkflow.ts` (task defined at line 483, so ~500 lines of setup
before the export) and `frontend/src/pages/PromptManagement.tsx` (7500+ lines, contains route-level
logic, prompt editing, and flow outlining). Neither is broken. Both are where the next bug will be
expensive.

### 4.2 Do the original decisions still hold?

| Decision | Problem it solved | Still a problem? | Verdict |
|---|---|---|---|
| **Supabase** (Postgres + edge fns + auth) | One vendor for DB, auth, RLS and serverless, with no ops | Yes | **Hold.** RLS is now load-bearing (GATE A) and is the tenant-isolation model. Replacing it means rebuilding auth, RLS and 97 functions. `SELF_HOSTING_FEASIBILITY.md` costs this honestly and its own recommendation is to spike the edge runtime first. Cost to replace: **very high**. |
| **Trigger.dev** for the cadence engine | Durable multi-day waits (`wait.until()`), resume-safe, retries | Yes, and it is the hardest part of the product | **Hold, reluctantly.** This is the deepest lock-in: the durable-wait semantics are the product. A BullMQ rebuild done wrong means double-sends or dropped leads to real leads. Cost to replace: **very high, highest risk of any component.** |
| **Retell** for voice | Managed LLM + TTS + telephony with mid-call tool calls | Yes | **Hold.** Replaceable in principle (5 tools behind one edge function), but the prompts are tuned in Retell's UI and are explicitly not repo-managed. Cost to replace: **medium-high, plus prompt re-tuning.** |
| **Twilio BYO per client** | Client is the legal sender under AU spam law; keeps BFD off the compliance hook | Yes, more than ever | **Hold. This is the best decision in the stack** and should not be revisited. |
| **GoHighLevel per client** | Clients already live in GHL; it is the CRM of the ICP | Yes | **Hold**, but it is the flakiest integration: no exposed merge tags forced the two-workflows-per-tenant hack in `WEBHOOKS.md` §G, and the RSA-not-HMAC webhook signing forced a static-token workaround. Highest ongoing friction per unit of value. |
| **OpenRouter** for LLM | One key, model-swappable | Yes | **Hold.** Genuinely low lock-in; the cheapest thing here to replace. |
| **n8n** (inherited) | Upstream's text engine | **No.** Solved and removed | **Correctly retired.** Verify the Railway service is actually switched off and stop paying for it. |
| **Railway** for the frontend | Cheap static-ish hosting, push-to-deploy | Yes | **Hold**, but note there is no Dockerfile or platform config in the repo, so the build contract is invisible and lives only in the Railway dashboard. That is a real bus-factor item. |

**The decision I would revisit:** the **per-client second Supabase project**. `ONBOARDING_GAP_REPORT`
and `FIRST_CLIENT_TASKS` both name it "the #1 un-automated blocker", and `.env.example` still carried
credentials for a `bfd-setter-live` project whose purpose was already flagged as unclear back in May.
It is manual, it is the slowest onboarding step, and the multi-DB design is what forces `types.ts` to
be hand-patched forever. Worth a hard look before client #2, not client #1.

### 4.3 Is the documentation system itself fit for purpose?

**The tracking half is genuinely excellent and I would not change it.** `SESSION_PLAN.md` plus the 6
canonical lists plus the Relay Protocol is a real, followed, working system. It survived this audit
with zero corrections, which almost nothing else did. The trigger phrases ("run test session", "I'm
onboarding a client") are a good idea.

**The descriptive half is where it fails, and the failure mode is specific: there is no owner and no
trigger for updating a doc when code changes.** The Relay Protocol mandates updating the *lists* at
closeout. It says nothing about updating `ARCHITECTURE.md` or `WEBHOOKS.md`, which is exactly why
those two drifted by two months while the lists stayed perfect. **This is the highest-leverage process
fix available: add "did this change make a living doc wrong?" to the Relay Protocol closeout.**

Other findings:
- **There is no single entry point.** `README.md`, `Docs/README.md`, `CLAUDE.md`, `AGENTS.md` and
  `SESSION_PLAN.md` all plausibly claim to be where you start, and each links to the others.
- **`CLAUDE.md` and `AGENTS.md` are maintained twins, which doubles the edit cost of every rule
  change** and means the two can silently diverge. They were in sync this time.
- **150 handoff files** (`Operations/handoffs/`, 38 of them) plus 30 archive docs plus the lists is a
  lot of surface for one person. The handoffs are valuable but there is no index.
- **Doc conventions are declared and not followed.** The workspace `CLAUDE.md` mandates YAML
  frontmatter with a `description` on every `.md`; most files in `Docs/` do not have it.

### 4.4 What would bite a new operator or a fresh AI session first?

In the order they would actually get burned:

1. **Reading a resurrected stale doc without knowing it.** Fixed this session, but the *cause* is not
   fixed: Syncthing will resurrect deleted files again. See §6.
2. **Believing a green `tsc` or a green `vite build`.** Both are meaningless here, and a GATE-A change
   white-screened production while both were green. Now documented in `CLAUDE.md`/`AGENTS.md`.
3. **Looking for edge functions in `supabase/functions/`** and concluding the repo is broken.
4. **Trusting a Retell agent's hardwired phone number** to tell you which agent is live. It does not;
   read `voice_setters.retell_agent_id`. This one has burned the project before
   (`MAIN-OUTBOUND-SHARED-1`).
5. **Editing a voice prompt.** Hard-forbidden and it silently reverts. Well documented already.
6. **Burning the harness refresh token on a "is it still valid?" probe.** Single-use; a probe kills
   the session. Documented, and it has happened.
7. **Deploying an edge function with the plain Supabase CLI**, which drops `_shared/` imports and
   broke 27 functions once already.
8. **Assuming `npm test` covers the edge functions' types.** It does not; `--no-check` and
   `--experimental-strip-types` both skip type checking entirely.

### 4.5 The single most important thing that is wrong or missing

**Committing to one answer: the project's real state lives in one person's head, and the repo's
documentation is a lagging, partial projection of it.**

Every concrete finding in this audit is a symptom of that. The lists are perfect because Brendan
touches them every session. `ARCHITECTURE.md` drifted two months because nothing forces it. The
handoffs are the real memory, and there are 38 of them with no index. The build contract lives in a
Railway dashboard. Which Retell agent is live cannot be answered from the repo at all. `bfd-setter-v2`
exists because reverse-engineering the product into a PRD was easier than reading its documentation.

This is not a documentation-quality problem, it is a **continuity risk**, and it is the thing most
likely to actually hurt the business. It becomes acute the moment there is a paying client whose
cadences must keep running while Brendan is unavailable.

### 4.6 Recommendations, ranked by leverage

1. **Add a doc-check to the Relay Protocol closeout.** One line: "did this session make
   `ARCHITECTURE.md`, `WEBHOOKS.md`, `CADENCE_DESIGN.md` or a SOP wrong? Fix it now." This is nearly
   free and prevents the exact drift this audit spent most of its time undoing.
2. **Stop Syncthing resurrecting deleted files.** Add the deleted paths to `.stignore`, or stop syncing
   `Projects/` and rely on git. Until this is fixed, this audit's cleanup will silently undo itself.
3. **Write the "if Brendan is unavailable" runbook.** One page: how to tell if cadences are running,
   how to stop them, who the live client is, where the secrets are, what to do if leads stop getting
   replies. This is the highest-value doc that does not exist.
4. **De-duplicate the Twilio signature helper** into `_shared/`. Small, contained, removes two copies
   of a security primitive. (A code change, therefore explicitly out of scope for this session.)
5. **Add `test:frontend`, `eslint` and `tsc -p tsconfig.app.json` to CI** as non-blocking reporting
   first, so the 21-error baseline becomes visible and cannot silently grow.
6. **Regenerate `types.ts`** to close the drift (`sms_delivery_events`, `client_cost_rollup`,
   `clients_public` columns). This alone would clear a chunk of the 21 type errors. Surgical adds only,
   per the multi-DB constraint.
7. **Decide `unipile-webhook` vs `receive-dm-webhook`** and delete the loser. Then confirm the Unipile
   signature scheme before arming any secret.
8. **Index the handoffs**, or fold the still-relevant ones into the living docs and archive the rest.

### H. Monthly running cost

**Largely UNVERIFIABLE FROM REPO, and that is itself a finding.** No document in this repository
records what the stack costs. Nothing was checked against a live billing dashboard, per scope. What
can be said from the repo:

- **Supabase** — free tier today. `DEFERRED.md` and `FIRST_CLIENT_TASKS.md` both gate HIBP on a "Pro
  upgrade" that has not happened, which implies the project is still on free. Pro is the known next
  step and the first real recurring cost.
- **Trigger.dev** — free tier, deliberately. Memory records a decision to stay free after the
  2026-06-23 latency incident was diagnosed as a regional degradation, not a concurrency limit.
- **Railway** — at least two services (frontend, n8n). **The n8n service is still being paid for while
  its code path is dead.** Shutting it off is pure savings.
- **Usage-metered and genuinely variable:** Twilio (per SMS and per voice minute, **billed to the
  client under BYO**), Retell (per voice minute), OpenRouter (per token). The first-client offer in
  memory prices a 1,500-minute pool, so voice minutes are the dominant variable cost.
- **GoHighLevel** — per client, client-owned.

**Best estimate: fixed platform cost is currently near zero and rises to roughly Supabase Pro plus
Railway on the first paying client, with true cost dominated by per-minute voice and per-token LLM
spend.** I will not put a dollar figure on it without the billing dashboards.

**Recommendation: record actual monthly cost somewhere in the repo.** The system has a
`client_pricing_config` table, `get-blended-rate`, `execution_cost_events` and a cost-to-price
calculator (F8), so it can compute what a client costs, but nobody wrote down what the platform costs.

### I. Known risks and failure modes

**Highest severity first.**

1. **Continuity / bus factor of one.** §4.5. The most likely thing to actually hurt the business.
2. **Syncthing silently resurrecting deleted files.** It has already happened once and put stale docs
   at canonical-looking paths. Unfixed, it will undo this audit.
3. **Verify-if-present webhooks are forgeable until armed.** 7 endpoints, including all 3 Retell
   receivers and 3 GHL receivers. Deliberate and tracked, but it is a live-traffic-affecting
   fail-open, and arming the secrets is precisely the step that can silently 403 all inbound traffic.
4. **Arming secrets can kill inbound traffic silently.** `WEBHOOKS.md` §E warns about this; get a
   known-good baseline first.
5. **`types.ts` drift.** Known, recurring (memory `feedback_types_ts_drift`), currently causing real
   type errors and one table the code writes that the types do not know about. The multi-DB design
   means it can never be wholesale-regenerated.
6. **No test above the unit line, and CI gates nothing.** A green CI run is compatible with a
   white-screened dashboard. This has happened.
7. **Cron auto-registration is not guaranteed on Trigger deploys.** Warned in source. A silent failure
   here stops the funnel refresh, the drift poll, and the synthetic probe, and nothing would alert you
   because the alerting task is itself one of the crons.
8. **Voice prompts are not in the repo and drift from it.** The repo's prompt files and
   `deploy_voice_prompt.mjs` still reference a **deleted** Retell LLM id. The repo cannot tell you what
   a live agent says.
9. **Shared-agent fan-out.** A smoke test once wiped a live LLM via shared-agent fan-out
   (`EE1`, 2026-05-18). Guarded since, but the class of bug is architectural.
10. **Duplicate GHL contacts.** Appointments attach by email, calls by phone, producing duplicates.
    Known, mitigated by disabling GHL allow-duplicates.
11. **The typecheck OOMs on this box.** Minor, but it means the one real check is the one most likely
    to be skipped.
